## Findings detail

### T-1 / P-1 🔴 `workspace:*` not replaced at publish time

**Affects:** `@tracelane/core`, `@tracelane/report`, `@tracelane/wdio`, `@peekdev/cli`, `@peekdev/mcp`

Every published package has literal `workspace:*` in its npm-published `dependencies`.
pnpm fails with `ERR_PNPM_WORKSPACE_PKG_NOT_FOUND`; npm fails with
`ERR_EUNSUPPORTEDPROTOCOL`. No user can do a fresh `pnpm install` or `npm install`
without this workaround.

Confirmed by running `npm view @tracelane/core@0.1.0-alpha.1 dependencies`:
```
{ '@cubenest/rrweb-core': 'workspace:*' }
```

**Root cause:** pnpm's workspace-protocol replacement did not run at publish time.
Check `publishConfig` and `.npmrc` — pnpm replaces `workspace:*` automatically only
when `pnpm publish` is run from the workspace root, not when publishing from a
package subdirectory with `--no-git-checks` or similar flags that bypass the lifecycle.

**Fix:** Republish all five packages after verifying `pnpm publish` resolves the
protocol. Run `npm view <pkg> dependencies` to confirm `workspace:*` is gone before
tagging.

**Workaround in QA workspace:** `pnpm.overrides` block in the fixture's `package.json`
and the QA workspace root `package.json`.

### P-2 🔴 `@peekdev/mcp` postinstall script not in `files`

`package.json` declares `"postinstall": "node ./scripts/postinstall-guard.mjs"` but
`"files"` is `["dist", "src/db/migrations", "src/native-host/extension-ids.json", ...]`
— `scripts/` is absent. Published package is missing the file, so postinstall crashes
with `MODULE_NOT_FOUND` on any fresh install.

**Fix:** Add `"scripts/postinstall-guard.mjs"` to `"files"`. Already done in source
at `packages/peek-mcp/package.json`, pending republish.

**Workaround in QA workspace:** `pnpm install --ignore-scripts`.

### P-3 🔴 `peek` CLI produces no output under pnpm bin shim

`invokedDirectly` check in `dist/index.js`:
```js
const invokedDirectly =
  process.argv[1] !== undefined && import.meta.url === `file://${process.argv[1]}`;
```

Under pnpm's virtual store, `import.meta.url` resolves to the deep
`.pnpm/@peekdev+cli@.../` path while `process.argv[1]` stays at the shallow
`node_modules/@peekdev/cli/dist/index.js` path (the pnpm-generated bin shim passes
the pre-symlink path). They never match → the CLI silently exits with code 0.

Same bug exists in `peek-mcp/src/postinstall.ts`.

**Fix:** Add `realpathSync` fallback:
```ts
import { realpathSync } from 'node:fs';
const invokedDirectly =
  process.argv[1] !== undefined &&
  (import.meta.url === `file://${process.argv[1]}` ||
    (() => { try { return import.meta.url === `file://${realpathSync(process.argv[1])}`; } catch { return false; } })());
```
Already done in source at `packages/peek-cli/src/index.ts` and
`packages/peek-mcp/src/postinstall.ts`, pending republish.

**Workaround in QA workspace:** `./peek` shim script that passes the resolved real
path to `node` directly.

### T-4 🟡 `TraceLaneService` TypeScript type error (`wdio.conf.ts` A.2)

`TraceLaneOptions` (the first constructor param) doesn't extend
`WebdriverIO.ServiceOption`, so `typeof TraceLaneService` is not assignable to
`ServiceClass`. `pnpm exec tsc --noEmit` on the fixture outputs:

```
wdio.conf.ts(43,5): error TS2322: Type '[typeof TraceLaneService, ...]' is not
assignable to type 'ServiceEntry'.
```

This blocks QA item A.2 from passing cleanly.

**Fix:** Change constructor signature in `packages/tracelane-wdio/src/service.ts`:
```ts
constructor(options: TraceLaneOptions & WebdriverIO.ServiceOption = {}, ...)
```
Already done in source, pending republish.

**Workaround in QA fixture copy and fixture source:** `// @ts-expect-error` comment
on the services tuple.

### T-7 🟡 Network capture unavailable in tracelane fixture (headless, no CDP)

Both specs log:
```
[tracelane/wdio] network capture unavailable (CDP not attached); degrading to rrweb+console only.
```

QA item C.4 requires the replay's network panel to show the `/api/will-fail` request.
Without CDP, the network panel will be empty. This degrades the headline-feature story
for a first-time user opening a report.

The fixture's `wdio.conf.ts` sets `capture: { network: true }` but doesn't register
`@wdio/devtools-service`, which is what attaches CDP in WDIO 9. The `@tracelane/wdio`
README should either document the devtools-service requirement or auto-register it
when `capture.network: true` is set.

**Fix options:**
1. Add `@wdio/devtools-service` to the fixture's devDependencies and register it in
   `wdio.conf.ts`; OR
2. Have `TraceLaneService` auto-require it when `capture.network: true`.
