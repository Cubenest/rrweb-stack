### P-10 ✅ RESOLVED (alpha.3) — `allowed_origins: []` in native host manifest — blocks all recording

Resolved by prompting for the unpacked extension ID during `peek init` and
overriding `extensionIds.dev` before `buildManifest()` runs (see
`packages/peek-cli/src/commands/init.ts`). The validator
(`packages/peek-cli/src/lib/extension-id.ts`) pins the 32-char a–p shape; an
empty submission is allowed (skip — only useful with a populated CWS slot).

Original report below:

**Observed:** After `peek init`, the written manifest has an empty `allowed_origins`:

```json
{
  "name": "com.cubenest.peek",
  "description": "peek local bridge — native messaging host for the peek browser companion",
  "path": "/opt/homebrew/lib/node_modules/@peekdev/cli/node_modules/@peekdev/mcp/dist/index.js",
  "type": "stdio",
  "allowed_origins": []
}
```

**Impact:** Chrome enforces that `allowed_origins` contains the extension's Chrome
Web Store ID (or unpacked dev ID) before allowing a native messaging connection. With
an empty array, Chrome silently blocks all `chrome.runtime.connectNative()` calls from
the extension. The extension's SW runs but cannot reach the native host → no events
are written to `~/.peek/sessions.db`. This is the root cause of all E.* and J.1–J.6
failures.

**Why counters update on reload only:** The extension buffers events locally in the
content script / SW. On page reload the tab's state resets, incrementing the counter.
But the flush to the native host is blocked, so nothing persists.

**Root cause in source:** `installManifests()` in
`packages/peek-mcp/src/native-host/installer.ts` writes `allowed_origins` but the
extension ID is not available at `peek init` time (the extension is loaded unpacked
and gets a locally-assigned ID that changes per machine). The correct approach is to
either:
1. Prompt the user for the unpacked extension ID during `peek init`, or
2. Populate `allowed_origins` with a known set of IDs (store ID + common dev IDs), or
3. Use the extension's `externally_connectable` manifest key as an alternative channel.

**Fix:** `peek init` must write a non-empty `allowed_origins`. At minimum it should
include the known dev extension IDs from `src/native-host/extension-ids.json`.

### T-9 ✅ RESOLVED (alpha.3) — rrweb recording empty: "Not enough recorded events to replay"

**Root cause:** the Service + hooks-factory were calling `session.onUrl` from
`beforeCommand`. WDIO fires `beforeCommand` BEFORE the actual command runs, so
the recorder was being re-injected on a page that was about to be torn down by
the navigation. The new page loaded with no recorder, no in-page buffer, and
the per-test poll harvested 0 events. Verified by running the e2e smoke spec
with `logLevel: 'info'` — `tracelaneInitScript` and `injectBundleScript` ran
once on `about:blank` (the pre-navigation page), then never again on the
loaded fixture.

**Fix:** moved the re-injection hook from `beforeCommand` to `afterCommand`
(see `packages/tracelane-wdio/src/service.ts` and
`packages/tracelane-wdio/src/hooks.ts`). `afterCommand` fires after WDIO
returns from `url()`, by which point Chrome has navigated and the new
document is loaded — `reinject()` now lands rrweb on the actual page under
test. Verified by decoding the e2e report blob: 15 events captured (Meta,
FullSnapshot, 9 interaction Incrementals, plus tracelane.nav boundaries).
Errors from the `url()` command itself skip re-injection (the navigation
never happened, so nothing changed).

### P-11 ✅ RESOLVED (alpha.3) — Side-panel event counters only update on page reload

**Root cause:** the SW only injected the MAIN-world recorder from
`chrome.tabs.onUpdated` with `status === 'loading'`. When the user clicked
"Enable for this site", `addEnabledOrigin()` persisted consent but no
existing tab got a recorder until the next reload — so counters stayed at
zero (in fact no events flowed to the SW at all; the on-reload bump was the
relay catching the page's initial DOM mutation burst, not user interaction).

**Fix:** added a `chrome.storage.onChanged` branch in
`packages/peek-extension/entrypoints/background.ts` that diff-checks the
`peek:enabledOrigins` key, queries every currently-open tab matching each
newly-added origin via `chrome.tabs.query({ url: '<origin>/*' })`, and runs
`maybeInject` on each. The `diffAddedOrigins` helper
(`packages/peek-extension/src/activation/storage.ts`) is the pure half;
unit-tested against junk payloads.

Original report below:

**Observed:** After enabling recording on a site, the event counters in the peek side
panel do not increment on user interaction (clicks, scrolls, input). They only jump
when the page is reloaded.

**Expected per D.3:** Counters should rise as the user interacts.

**Likely cause:** The content script or SW is batching events but the side panel UI
is not receiving live updates — either the `port.postMessage` from the content script
to the side panel is not firing on interaction, or the side panel's listener is only
wired to the `onActivated` / `onUpdated` tab events (which fire on reload).

**Note:** Even if the UI were fixed, data still can't flush to the DB due to P-10
(`allowed_origins: []`). Both bugs need fixing for E.* to pass.

### P-14 🔴 NEW — `debugger` in `optional_permissions` rejected by Chrome MV3

**Observed:** Loading the extension in Chrome 148 shows:
> "Permission 'debugger' cannot be listed as optional. This permission will be omitted."

**Root cause:** Chrome 121+ removed `debugger` from the allowed set of optional
permissions in Manifest V3. The `optional_permissions: ["debugger"]` entry in
`manifest.json` is silently dropped at load time.

**Impact:**
- C.4 fails — QA doc expects `debugger` under optional permissions; Chrome won't grant it
- Group H (Deep Capture) is entirely blocked — `chrome.debugger.attach()` requires
  the permission to be declared, but it can't be optional in MV3

**Fix options:**
1. Move `debugger` to required `permissions` — triggers install warning but works
2. Remove it entirely and document Deep Capture as unsupported in this build
3. Use `chrome.debugger` via a different mechanism (e.g. Offscreen document with a
   Manifest V2 component — complex)

Option 1 is the pragmatic path for alpha. A required `debugger` permission with clear
justification text is better than silently broken Deep Capture.

### P-16 🔴 NEW — Native host crashes: Chrome uses wrong Node.js (v14 x86_64 vs arm64)

**Observed:** `peek sessions list` always returns empty despite the manifest being
correctly written, the extension running, and the user actively browsing.

**Root cause:** The native host manifest `path` field points directly to the `.js`
binary (`dist/index.js`, shebang `#!/usr/bin/env node`). On this machine:
- `/usr/local/bin/node` = v14.21.2, x86_64 (legacy install from Dec 2022)
- `/opt/homebrew/bin/node` = v24.4.1, arm64 (the correct runtime)

Chrome and all macOS GUI apps launched from the Dock use the system `$PATH`
(`/usr/local/bin:/usr/bin:/bin:...`). `env node` resolves to `/usr/local/bin/node`
(v14, x86_64). This binary cannot load `better-sqlite3` which was compiled for
arm64, so the native host exits immediately with:

```
Error: dlopen(...better_sqlite3.node): mach-o file, but is an incompatible
architecture (have 'arm64', need 'x86_64')
```

Chrome receives no response, silently moves on. The SW's `connectNative` call is
accepted (manifest is valid) but the host process dies before sending any data.

**Fix (source):** In `packages/peek-mcp/src/native-host/installer.ts` /
`packages/peek-cli/src/commands/init.ts`, write a shell wrapper at
`~/.peek/peek-mcp-host.sh` using `process.execPath` (the node that ran
`peek init`) and point the manifest `path` to the wrapper, not the raw `.js`:

```sh
#!/bin/sh
exec /opt/homebrew/bin/node /path/to/dist/index.js "$@"
```

This is a well-known macOS native messaging issue — any native host that needs a
non-system runtime must use a wrapper script.

**Manual workaround for QA:**
```sh
cat > ~/.peek/peek-mcp-host.sh << 'EOF'
#!/bin/sh
exec /opt/homebrew/bin/node \
  /opt/homebrew/lib/node_modules/@peekdev/cli/node_modules/@peekdev/mcp/dist/index.js "$@"
EOF
chmod +x ~/.peek/peek-mcp-host.sh
# Update all manifests to point to the wrapper:
for f in \
  ~/Library/Application\ Support/Google/Chrome/NativeMessagingHosts/com.cubenest.peek.json \
  ~/Library/Application\ Support/Chromium/NativeMessagingHosts/com.cubenest.peek.json \
  ~/Library/Application\ Support/Microsoft\ Edge/NativeMessagingHosts/com.cubenest.peek.json; do
  python3 -c "import json,sys; d=json.load(open(sys.argv[1])); d['path']='$HOME/.peek/peek-mcp-host.sh'; json.dump(d,open(sys.argv[1],'w'),indent=2)" "$f" 2>/dev/null
done
```
Then reload the extension in Chrome (↺ — same ID).
